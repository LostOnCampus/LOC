## 슬라이드 5: 영역 Ⅲ — LiteLLM 활용 및 AI 기반 개발 과정

- 배점: 25점
- 영역 성격: 핵심 영역
- 평가 관점: LiteLLM을 개발 파트너로 활용했는가
- 핵심: 단순 코드 생성이 아니라 요구사항 정의 → 생성 → 검토 → 오류 해결 → 개선의 반복 과정
- 원문 캡션: Area Ⅲ · 다음 장에서 25점 수준별 채점 기준 상세

| 항목 | 배점 | 평가 항목 | 평가 기준 |
|---|---:|---|---|
| 항목 01 | 5 | 웹 소스 생성 | LiteLLM으로 HTML, CSS, PHP, SQL 소스를 생성한 기록 |
| 항목 02 | 5 | 프롬프트 설계 능력 | 기능 요구사항, DB 구조, 화면 구성, 오류 조건을 명확히 제시 |
| 항목 03 | 5 | 생성 코드 검토 및 수정 | AI 코드를 그대로 제출하지 않고, 팀 목적에 맞게 수정·통합 |
| 항목 04 | 4 | 오류 해결 과정 | Apache·PHP·DB 연결 오류를 LiteLLM에 질문하고 해결 |
| 항목 05 | 3 | 기능 개선 반복 과정 | 생성 → 실행 → 오류 확인 → 재질문 → 수정의 반복 |
| 항목 06 | 3 | 활용 기록 정리 | 프롬프트, 생성 코드, 수정 전후 차이, 최종 반영을 보고서에 기록. 원문은 `보고서에`에서 문장이 끝남 |

## 슬라이드 6: 영역 Ⅲ 상세 — LiteLLM 25점 수준별 채점 기준

> LiteLLM 활용, 어디까지 갔는가 — 최우수팀은 LiteLLM을 개발 파트너처럼 활용한 팀

| 수준 | 특징 | 점수 범위 |
|---|---|---:|
| 미흡 | LiteLLM 활용 기록이 거의 없고, 단순 복사 수준 | 0-8점 |
| 보통 | LiteLLM으로 기본 코드 생성은 했으나 수정·검토 과정이 약함 | 9-15점 |
| 우수 | 프롬프트를 구체화하고, 생성 코드를 수정하여 웹사이트에 반영 | 16-21점 |
| 최우수 | 오류 해결, 기능 개선, 코드 구조화까지 반복 활용하고 그 과정을 설명 가능 | 22-25점 |

> "코드 한 번 뽑았다"가 아니라 "파트너처럼 활용했다"를 평가합니다.
